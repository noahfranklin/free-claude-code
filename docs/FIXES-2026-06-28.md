# Fixes — 2026-06-28 (version 2.5.0)

Engineering write-up for the three reliability fixes shipped in this fork
(version **2.5.0**, forked from upstream **2.3.20**). Each section describes the
symptom, the root cause and the modules involved, the fix, and how to verify it.

---

## 1. Stream idle-timeout watchdog

### Symptom

Claude Code (or Codex) would appear to "hang" for roughly five minutes with no
output. This was most visible when Claude Code auto-selected a non-responding
NVIDIA NIM model: the upstream accepted the HTTP request, returned headers, and
then sent no body bytes. The client sat waiting until the full 300-second read
timeout finally tripped.

### Root cause

The proxy only enforced an overall read timeout on the upstream connection.
Connecting and receiving response headers counts as activity, so once the
upstream stalled mid-stream there was no shorter deadline to detect the silence.
The ~300s read timeout was the only backstop, producing a long silent hang
instead of a fast, actionable error.

### Fix

A dedicated idle watchdog (`core/anthropic/stream_watchdog.py`) wraps the
upstream byte stream. It resets a timer on every chunk received; if no bytes
arrive within `HTTP_STREAM_IDLE_TIMEOUT` (default 60s), it aborts the upstream
and raises a clean Anthropic-shaped error that propagates back to the client as
a normal error event rather than a dangling connection. The overall read timeout
remains as a longer outer bound; the watchdog catches the common "accepted but
silent" case far sooner.

### How to verify

1. Set `MODEL` (in the Admin UI) to a model known not to respond, or point a
   provider base URL at an endpoint that accepts the connection but never streams.
2. Run `fcc-claude` and send a prompt.
3. Confirm the request aborts in roughly `HTTP_STREAM_IDLE_TIMEOUT` seconds
   (about 60s by default) with a clean Anthropic error, not after ~300s.
4. Lower `HTTP_STREAM_IDLE_TIMEOUT` (e.g. to `5`) to make the abort fast and
   confirm the timing tracks the setting.

## 2. Clean missing-key error (Anthropic 401)

### Symptom

Selecting a provider that had no API key configured produced a malformed HTTP
500 from the proxy. The client received an opaque server error with no guidance
on what was wrong or how to fix it.

### Root cause

The missing-credential condition was not translated into the Anthropic error
schema before reaching the response layer, so it fell through as an unhandled
500 instead of an authentication error the client could display meaningfully.

### Fix

Credential resolution in `api/dependencies.py` now raises a typed provider error
when a provider has no API key, and the app-level `ProviderError` handler maps it
to a proper Anthropic `401 authentication_error`. The error message names the
expected environment variable for that provider and points to where a key can be
obtained, so the failure is self-explanatory in the client.

### How to verify

1. In the Admin UI, select a provider but leave its API key blank (e.g. clear
   `NVIDIA_NIM_API_KEY` while `MODEL` targets that provider).
2. Send a request through `fcc-claude` or `curl` to `/v1/messages`.
3. Confirm the response is HTTP `401` with an Anthropic
   `authentication_error` body that names the missing env var, not a 500.

## 3. Working-models-only model picker (model-health tracking)

### Symptom

The `/v1/models` picker listed models that were not actually usable — models that
consistently errored or timed out still appeared, so users kept selecting dead
models from the native `/model` picker.

### Root cause

The model catalog was static with respect to runtime health: there was no signal
recording which models had recently failed, and no filtering of the listing based
on that signal.

### Fix

Two cooperating mechanisms were added:

- **Reactive auto-demotion.** Health state is tracked in `api/model_health.py`.
  Request execution in `api/provider_execution.py` reports failures (errors and
  timeouts) into the health store. The catalog builder in `api/model_catalog.py`
  consults this state and filters the `/v1/models` listing according to
  `FCC_MODEL_LIST_MODE`. A demoted model is hidden until `FCC_MODEL_HEALTH_COOLDOWN`
  elapses, after which it can reappear and prove itself again.
- **Proactive health-check.** The Admin UI exposes a **"Check working models"**
  button backed by endpoints in `api/admin_routes.py`. It probes models across all
  credentialed providers concurrently and refreshes health state up front, rather
  than waiting for organic traffic to surface failures.

When `FCC_MODEL_HEALTH_ENABLED=false` or `FCC_MODEL_LIST_MODE=all`, behavior
matches upstream: the full catalog is listed.

### Endpoints

- `GET /admin/api/models/health` — current health state for tracked models.
- `POST /admin/api/models/health-check` — run a proactive probe across credentialed
  providers and update health state.

### How to verify

1. Ensure `FCC_MODEL_HEALTH_ENABLED=true` and set
   `FCC_MODEL_LIST_MODE=exclude_unhealthy` (or `healthy_only`).
2. Use a model that fails (see fix #1's non-responding model). After it errors or
   times out, request `/v1/models` and confirm the failing model is no longer
   listed (reactive demotion).
3. Wait `FCC_MODEL_HEALTH_COOLDOWN` and confirm the model can reappear.
4. In the Admin UI, click **"Check working models"** (or
   `POST /admin/api/models/health-check`) and confirm the listing reflects only
   working models. Inspect `GET /admin/api/models/health` for per-model state.

## Configuration

All new environment variables and their defaults:

| Env var | Default | Meaning |
| --- | --- | --- |
| `HTTP_STREAM_IDLE_TIMEOUT` | `60` | Seconds of upstream stream silence (no bytes received) before the watchdog aborts the request with a clean Anthropic error. |
| `FCC_MODEL_HEALTH_ENABLED` | `true` | Master switch for model-health tracking and health-aware `/v1/models` filtering. When `false`, the full catalog is listed. |
| `FCC_MODEL_LIST_MODE` | `exclude_unhealthy` | How `/v1/models` filters by health: `all` (no filtering), `exclude_unhealthy` (hide demoted models), `healthy_only` (list only models confirmed healthy). |
| `FCC_MODEL_HEALTH_COOLDOWN` | `300` | Seconds a model stays demoted after a failure before it may reappear in the listing. |
| `FCC_MODEL_HEALTH_PROBE_TIMEOUT` | `30` | Per-model timeout (seconds) for the proactive "Check working models" probe. |
| `FCC_MODEL_HEALTH_PROBE_CONCURRENCY` | `4` | Maximum number of model probes run concurrently during a proactive health-check. |

> Defaults shown reflect the documented design intent. Confirm exact values
> against `.env.example` and the settings schema, which are the source of truth.

## Design note: the real `claude` is never modified

FCC does not patch or wrap the official `claude` binary in place. It ships
separate launchers — `fcc-claude` and `fcc-codex` — that run the real CLI with
`ANTHROPIC_BASE_URL` pointed at the local proxy. Your normal `claude` session
(Pro/Max or the Anthropic API) is unaffected and continues to talk to Anthropic
directly.
